# 1 Введение

<a name="appointment"/>

## 1.1 Назначение

Данный документ описывает функциональные и нефункциональные требования к веб-приложению «Менеджер семейного бюджета», адаптированные под структуру и технологический стек репозитория `laba2` (TypeScript/JavaScript фронтенд). Документ служит основой для разработки, тестирования и приемки приложения.

> Примечание: при обзорe репозитория `laba2` замечено использование TypeScript/JavaScript; поэтому приложение спроектировано как веб-клиент (React/TypeScript или аналог) с возможностью офлайн-работы (PWA). ·citeturn1view0

<a name="business_requirements"/>

## 1.2 Бизнес-требования

<a name="initial_data"/>

### 1.2.1 Исходные данные

Семьи нуждаются в простом, удобном инструменте для планирования и контроля семейных финансов. Веб-версия приложения должна быть доступна с разных устройств (десктоп, мобильный браузер), при этом предусматривать работу офлайн и хранение данных локально, чтобы не требовать постоянного подключения к сети.

<a name="business_opportunities"/>

### 1.2.2 Возможности бизнеса

Продукт помогает пользователям быстро регистрировать доходы и расходы, ставить бюджетные цели и получать визуальные отчеты. Легкая доступность через браузер и отсутствие обязательной подписки расширяют аудиторию. Возможна монетизация через премиум-функции (резервное шифрованное хранилище, облачная синхронизация) — не обязательна для первой версии.

<a name="project_boundary"/>

### 1.2.3 Границы проекта

Приложение реализует:

* пользовательский интерфейс в браузере (SPA на TypeScript/JS);
* локальное хранение данных (IndexedDB / localStorage) и офлайн-режим через Service Worker (PWA);
* CRUD для транзакций, категорий и бюджета;
* базовую аналитику и графики;
* экспорт/импорт данных в CSV.

Не входит в сферу первой версии: интеграция с банковскими API, платная подписка, серверная часть с аккаунтами в облаке (опционально как расширение).

<a name="analogues"/>

## 1.3 Аналоги

Аналогами являются веб- и мобильные версии Moneon, YNAB, Cash Organizer. От них берем идеи UX (категории, быстрый ввод транзакции, сводная панель), но упрощаем функционал для быстрого старта и офлайн-доступности.

<a name="user_requirements"/>

# 2 Требования пользователя

<a name="software_interfaces"/>

## 2.1 Программные интерфейсы

* Локальное хранилище: IndexedDB (рекомендуется через библиотеку idb) для основной информации; fallback в localStorage для небольших данных.
* PWA: Service Worker для офлайн-кеша и синхронизации.
* (Опционально) REST API для синхронизации с сервером, если будет реализована облачная синхронизация.

<a name="user_interface"/>

## 2.2 Интерфейс пользователя

Приложение — SPA с адаптивным интерфейсом. Основные экраны:

* Экран приветствия / входа (имя пользователя / локальная учётная запись);
* Главная панель (dashboard) — текущее состояние бюджета, быстрые действия;
* Форма добавления транзакции;
* Список транзакций и фильтры по категории/периоду;
* Экран категорий;
* Экран аналитики (диаграммы по категориям, динамика доходов/расходов);
* Экран профиля и экспорта/импорта данных.

Макеты и компоненты следует хранить в `src/components` и `src/pages` (если используется React/Next/CRA).

<a name="user_specifications"/>

## 2.3 Характеристики пользователей

<a name="user_classes"/>

### 2.3.1 Классы пользователей

| Класс пользователей                    | Описание                                                                                                 |
| :------------------------------------- | :------------------------------------------------------------------------------------------------------- |
| Локальные пользователи                 | Пользователи, создавшие локальный профиль в приложении (имя). Имеют доступ к полному функционалу офлайн. |
| Профили с синхронизацией (опционально) | Если реализована облачная синхронизация — пользователи с учётной записью на сервере.                     |

<a name="application_audience"/>

### 2.3.2 Аудитория приложения

<a name="target_audience"/>

#### 2.3.2.1 Целевая аудитория

Семьи и пары, желающие удобно вести совместный бюджет через браузер и мобильные устройства.

<a name="collateral_audience"/>

#### 2.3.2.2 Побочная аудитория

Индивидуальные пользователи, желающие простое решение для учёта личных расходов.

<a name="assumptions_and_dependencies"/>

## 2.4 Предположения и зависимости

* Приложение запускается в современных браузерах (Chrome, Edge, Firefox, Safari mobile/desktop).
* Для сборки проекта требуется Node.js (версия 16+ рекомендуема) и менеджер пакетов (npm/yarn).
* Если используется React + TypeScript — соответствующая конфигурация в репозитории.

<a name="system_requirements"/>

# 3 Системные требования

<a name="functional_requirements"/>

## 3.1 Функциональные требования

<a name="main_functions"/>

### 3.1.1 Основные функции

<a name="user_logon_to_the_application"/>

#### 3.1.1.1 Вход пользователя в приложение

| Функция                        | Требования                                                                                               |
| :----------------------------- | :------------------------------------------------------------------------------------------------------- |
| Регистрация локального профиля | Пользователь вводит имя для создания профиля, данные сохраняются локально.                               |
| Проверка уникальности имени    | Если локальный профиль с таким именем уже существует — показать сообщение и предложить другое имя/войти. |
| Вход                           | Позволить выбрать существующий локальный профиль для работы.                                             |

<a name="budget_managment"/>

#### 3.1.1.2 Управление бюджетом

Пользователь может задать семейный бюджет и период (день/неделя/месяц/год).

| Функция                    | Требования                                                |
| :------------------------- | :-------------------------------------------------------- |
| Установка значения бюджета | Поле ввода суммы и периода; валидация числового значения. |
| Выбор единиц периода       | Выпадающий список: день, неделя, месяц, год.              |

<a name="budget_changes"/>

#### 3.1.1.3 Изменение бюджета

| Функция                   | Требования                                                                                       |
| :------------------------ | :----------------------------------------------------------------------------------------------- |
| Создание транзакции       | Форма ввода: сумма, категория, тип (доход/расход), дата, комментарий. Сохранение в локальную БД. |
| Редактирование транзакции | Возможность редактировать ранее добавленную запись.                                              |
| Удаление транзакции       | Подтверждение перед удалением. Возможность восстановления через экспорт/импорт (или корзину).    |

<a name="cost_and_income_analytics"/>

#### 3.1.1.4 Аналитика расходов и доходов

| Функция                     | Требования                                          |
| :-------------------------- | :-------------------------------------------------- |
| Просмотр истории транзакций | Фильтры по периоду и категории, поиск по описанию.  |
| Статистика по категориям    | Круговые/столбчатые диаграммы, проценты от бюджета. |
| Динамика                    | График изменения расходов/доходов по времени.       |

<a name="user_log_out"/>

#### 3.1.1.5 Выход зарегистрированного пользователя из учётной записи

| Функция                          | Требования                                |
| :------------------------------- | :---------------------------------------- |
| Смена локального профиля / Выход | Возможность выйти в экран выбора профиля. |

<a name="restrictions_and_exclusions"/>

### 3.1.2 Ограничения и исключения

1. Первая версия не требует интернет-соединения (off-line-first);
2. Фокус на совместном/семейном бюджете, а не на банковских интеграциях;
3. Отсутствие серверной авторизации в первой версии.

<a name="non-functional_requirements"/>

## 3.2 Нефункциональные требования

<a name="quality_attributes"/>

### 3.2.1 Атрибуты качества

<a name="requirements_for_ease_of_use"/>

#### 3.2.1.1 Требования к удобству использования

1. Интуитивно понятный интерфейс; минимальное число кликов для добавления транзакции;
2. Кнопки и поля имеют понятные подписи;
3. Адаптивная верстка для мобильных и десктопов.

<a name="security_requirements"/>

#### 3.2.1.2 Требования к безопасности

1. Данные пользователя хранятся локально; при реализации облачной синхронизации — данные должны быть переданы по HTTPS и храниться в зашифрованном виде;
2. При экспорте данных — предупреждение о незашифрованном CSV.

<a name="access_requirements"/>

#### 3.2.1.3 Требования к доступности

1. Контрастность элементов согласно WCAG AA;
2. Навигация с клавиатуры и поддержка экранных читалок (aria-атрибуты);
3. Увеличенные размеры элементов для пользователей старшего поколения.

<a name="external_interfaces"/>

### 3.2.2 Внешние интерфейсы

* Экранные размеры: мобильные телефоны (min-width 320px), планшеты и десктопы;
* Поддерживаемые браузеры: Chrome (последние 2 версии), Firefox (последние 2), Edge, Safari (iOS).

<a name="restrictions"/>

### 3.2.3 Ограничения

1. Платформа — веб (SPA) — приложение не завязано на Android или Kotlin;
2. Язык реализации — TypeScript/JavaScript (как в репозитории `laba2`).
